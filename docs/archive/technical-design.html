技术设计文档 · Technical Design

语义 BOM 规则编译器
技术设计规范

面向实现的完整设计：节点架构、契约 schema、MySQL 数据模型、数据流、各模块算法逻辑、技术栈与分期。开发照此落地，业务/评审照此对齐。

版本V0.1日期2026-06-28形态Python · Vue3 · FastAPI · MySQL解耦独立于抽取平台

01系统总览

一句话：从测试集生成/优化"语义BOM"（定义+规则+召回画像），版本化管理，对导入的跑批结果做归因与收益分析。与抽取平台解耦，靠 Excel 导入导出桥接。

1.1 边界

本工具负责抽取平台负责（不在本项目）
BOM 生成/优化、诊断归因、版本管理、脱敏、大盘分析文档解析、召回、抽取、跑分（产出准确率/badcase）

1.2 设计目标

  • 解耦：不直连平台，通过导入导出同步，降低耦合与合规风险。
  • 契约先行：每个节点输入输出 Pydantic 固定，可独立测试、可并行开发。
  • 可信度分层：事实/计算(🟢)直接示人；AI 生成(🟡)必标"待复核"；阈值分级(🔴)待领导定。
  • 配置外置：提示词、模型、脱敏规则全在文件，代码零硬编码。

02总体架构（节点 skill 流水线）

每个节点是一个有输入输出契约的 skill，可独立测试、可替换。未来对接平台时，对接点=节点边界。

ingest 数据加载→anonymize 脱敏→generate 生成(Stage1+2)→diagnose 诊断→optimize 优化→verify 自检(Stage3)→apply 版本化→evaluate 大盘

2.1 三套配置页签（对齐新平台）

页签内容位置

03节点契约（Pydantic schema）

契约是协作边界。A 实现 generate、B 实现 diagnose，彼此只看对方契约。改契约=开 PR。下列为核心契约（开发第一步即落地 src/cc_bom_generator/contracts/）。

3.1 TestSet（数据加载输出）

# 一个文档 → 多语义块 → 多语义项 + 上下文 TestSet: documents: List[Document] Document: doc_id: str doc_name: str blocks: List[Block] Block: block_code: str # 语义块编码（父） block_name: str context: str # 上下文(条款期望值 或 chunk) items: List[Item] # 语义项（子），可空 Item: item_code: str item_name: str expected: str # 期望值，空=负例

3.2 BOM（生成/优化产物，版本化存储）

BOM: clause: str block_code: str version: int semantic_definition: str # 含【包含】【不含】边界 extraction_rules: ExtractionRules recall_profile: RecallProfile ExtractionRules: absolute_interception_rules: List[{rule:str, fixes:str}] # 针对误抽 core_match_rules: List[{rule:str, fixes:str}] # 针对漏抽 RecallProfile: positive_keywords: List[str] # 短词,程序过滤 confusion_words: List[str] section_hints: List[str] semantic_queries: List[str] # 定义改写,向量召回 positive_examples: List[str] # 多样性挑选

3.3 Diagnosis / Verification（诊断与自检）

Diagnosis: # 🟡 AI 归因 case_id: str case_type: "miss"|"false_positive" category: "召回|混合|BOM|Prompt模板|推理" reason: str suggested_fix: str fix_target: "rules"|"recall_profile"|"both" # 路由到Stage2 Verification: # 🟡 Stage3 自检 checks: List[{item:str, verdict:"pass|warn|fail", reason:str}] red_flags: List[str] coverage_estimate: str # 如 5/6≈83% summary: str

3.4 RunResult / Metrics（平台导入 + 本工具计算）

RunResult: # 🟢 平台跑批导入 block_code: str; version: int accuracy: float; miss: int; false_positive: int badcases: List[Badcase] OptGain: # 🟢 本工具算 fixed: int; regressed: int; net: int regression_guard: List[str] # 被改坏的其他条款

04数据模型（MySQL + JSON 列）

结构化元数据用关系表；BOM/trace 等嵌套结构用 JSON 列。schema 由 SQLAlchemy + Alembic 管理，变更提交 git，各人 alembic upgrade head 同步。

表关键字段说明
clauseblock_code(PK), block_name, domain, current_version条款注册（采购/销售/服务…）
bom_versionid, block_code, version, bom_json(JSON), parent_version, source(generate/optimize), created_by, created_at每版 BOM 快照（不可变）
testsetid, block_code, kind(initial/run_result), data_json, imported_at测试集/跑批原始数据
run_resultid, block_code, version, accuracy, miss, false_positive, imported_at🟢 平台导入的跑分
badcaseid, block_code, version, doc_id, type, expected, actual, similarity错例（是否匹配=否）
diagnosisid, badcase_id(FK), category, reason, suggested_fix, fix_target🟡 AI 归因结果
optimization_logid, block_code, from_ver, to_ver, fixed, regressed, net, operator, created_at🟢 优化收益记录
traceid, badcase_id(FK), trace_json, parsed_jsontrace 原文+解析后
chat_session/message—对话（Phase 2）
字符集

库与表统一 utf8mb4 / utf8mb4_unicode_ci（合同全中文）。JSON 列用 MySQL 原生 JSON 类型存嵌套 BOM。

05数据流与解耦（导入导出桥接）

规则生成与抽取是两套系统。本工具是"分析+版本+大盘"层，原料（准确率/badcase）由平台导出后导入。

① generate（初始）

初始测试集(期望值) → ingest → [脱敏] → generate(三阶段) → BOM v1 → 导出给平台配置

② optimize（已跑）

平台跑批 → 导出 Excel(期望/实际/是否匹配) → 导入 run_result+badcase → diagnose(吃trace) → optimize → BOM v2 → 导出

③ evaluate（大盘）

多版本 run_result 导入 → 算 修复/回归/净收益/归因占比 → 大盘展示 → 驱动下一轮

关键约束

没有"导入的跑批结果"就算不出收益/回归/准确率趋势——这些指标强依赖平台导出。

06三阶段流程（与平台逻辑对齐）

顺序：Stage1 定义+规则 → Stage2 召回画像(派生) → Stage3 自检。画像依赖定义，不是相反。

阶段产出温度可信度
Stage1 定义+规则semantic_definition + extraction_rules（+optimize 时 diagnosis）0.2 收敛🟡 AI
Stage2 召回画像recall_profile（从 Stage1 派生）+ derivation_check0.5 发散🟡 AI
Stage3 规则自检verification（红旗+覆盖估计）0.0🟡 AI（第二意见）

两场景共用三阶段：generate 无 badcase（Stage1 不诊断）；optimize 吃 badcase+trace（Stage1 先诊断 fix_target，Stage2 落实 recall_profile/both 的修复）。

07召回画像生成逻辑

  • 派生约束：positive_keywords 取自定义领域词；confusion_words 来自定义"不含"侧与拦截规则；semantic_queries 是定义改写；positive_examples 从候选多样性挑选。
  • 关键词硬过滤（程序化）：丢弃含数字/金额/百分比/公司名/整句标点/超长(中文>8字)的项——防过拟合。🟢 确定性
  • 候选去重：difflib 相似度贪心(阈值 0.8) + 打印"去掉哪些/组装多少"。🟢（Phase 2 升级 embedding+MMR）
  • 数量/语言可配：keyword_count/section_count/query_count/language_hint。

08诊断与自检逻辑

8.1 Badcase 诊断（5 类归因）🟡

依据 trace 证据归因（无 trace 只判症状、低置信）：

  • 原文窗口无目标文本 → 召回问题（改画像）
  • 窗口有但漏抽/误抽 → BOM 问题（改规则）；覆盖率缺口特判：检查"剔除规则"是否误杀期望段
  • reasoning 显示模型想错 → 推理问题（补 few-shot）
  • 提示词 I/O 缺陷 → Prompt 模板；多因 → 混合

每条带 fix_target 路由修复落点。阈值固定 80%，靠"抽得更完整"达标。

8.2 规则自检 Stage3 🟡

审核员拿 BOM+正例+误抽反例逐条验证：正例能否抽(会不会被拦截误杀)、反例能否拦、覆盖估计、红旗（拦截误杀/写死公司名=过拟合）。是"第二意见"，Phase 2 可升级为确定性规则引擎 🟢。

09脱敏引擎

  • 定位：出网合规闸门。业务在工具内看真实数据；仅数据出网去外部 LLM 时瞬时假名化，不持久化。
  • 策略：角色化占位（【采购方】【供应商】【金额_1】【人名_1】），保句法结构；规则(手机/身份证/金额)确定性 🟢，NER(公司/人名) 🟡。
  • 可逆映射：每次调用内存内维护，返回规则可回贴（角色化本身通用，不必反查）。
  • 规则可配：对接公司合规清单，设置页管理。

10版本管理 / 收益 / 回归

  • 版本快照：每次生成/优化存 bom_version（不可变）；可回滚、可 diff。🟢
  • 优化收益：版本 N vs N-1 两份导入的跑批结果做 badcase 集合 diff：旧错→新对=修复；旧对→新错=回归；net=修复-回归。🟢
  • 回归守护：优化条款 A 后，把其它条款的跑批结果也导入对比，看准确率是否下降（灾难性遗忘）。🟢（前提：其它条款结果也导入）
可信度说明

收益/回归是确定性计算(🟢)，但前提是导入的跑批结果准确；导入错则结论错。

11大盘指标逻辑（哪些事实、哪些 AI）

指标来源可信度
整体准确率 / 趋势导入 run_result 聚合🟢 事实
修复 / 回归 / 净收益版本 diff 计算🟢 计算
badcase 5 类归因占比依赖 diagnosis🟡 继承 AI 归因
重点关注(最低/下降最多)事实排序🟢 事实排序
测试集建议LLM 分析(仅低准触发)🟡 建议
不预设评级

大盘不设"优良中差"健康度；任何分级阈值由业务领导定义后才上线。仅展示事实数据与事实排序。

14分期路线

期范围
Phase 1（PoC 已验 + 核心）ingest/generate/optimize/verify/脱敏/版本/基础大盘；API+手动双模式；Excel 导入导出
Phase 2向量库(历史 few-shot + 多样性正例 embedding)、对话 Copilot Agent、回归守护自动化、多用户
Phase 3部署、与平台对接（对接点=节点边界）、权限

15配置体系（全外置）

  • 模型：config/llm.yaml（api_key/base_url/model/分阶段温度）。
  • 提示词：prompts/*.txt（XML 结构 + 每阶段角色 + {{var}} 占位），改提示词不改代码。
  • 脱敏规则：设置页 + 配置文件。
  • skill 参数：去重阈值、画像数量、trace_fields、language_hint 等 per-条款 yaml。

16非功能

  • 数据隔离：各开发本地 MySQL，数据不入库；样例须脱敏。
  • 合规：出网自动脱敏；真实合同数据绝不直接发外部模型。
  • 性能：单机/少数人 PoC，SQLite 级并发足够；Phase 3 多用户再优化。
  • 可观测：每节点日志（去重明细、trace 解析、归因）；AI 产物必标可信度。
