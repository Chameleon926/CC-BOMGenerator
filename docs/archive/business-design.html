语义 BOM 规则编译器
与自动调优平台

一个独立于新旧主平台的「规则实验室」——让合同商务人员用最少的操作，完成语义 BOM 的生成、Badcase 智能归因、规则调优与回归评估，闭环可视。

版本V0.1（设计评审稿） 日期2026-06-26 形态外网 Python · 工业级 Demo 受众业务 / 产品 / 评审

01项目背景与痛点

合同要素抽取是合同商务的核心环节，但围绕 Badcase 的诊断与调优链路，至今仍是业务人员的负担。

1.1 现状：新旧平台并存

当前合同要素抽取存在两套平台，调优理念截然不同：

旧

旧平台 · 提示词工程

依赖长文本 Prompt。遇 Badcase 要复制完整提示词给大模型做「黑盒」优化，返回整段新 Prompt。

新

新平台 · 规则声明式

将抽取解耦为「召回 + 抽取」，用户只维护语义 BOM 定义（业务定义 + 抽取/拦截规则 + 召回画像），系统自动拼装 Prompt。

1.2 业务人员的三大痛点

01

诊断链路长、手工分析重

Badcase 出现后，要进到 Trace 里手工逐条分析：是召回没命中、是 BOM 定义问题，还是大模型推理出错？人肉定位、人肉归类，耗时且依赖经验。

02

提示词调优门槛高、复制粘贴繁

Badcase 多时要一个个复制信息（误抽/漏抽、原文、reasoning）去问外部大模型优化；优化完还要重新生成抽取 BOM，反复横跳，极易出错。

03

回归测试难、对比不可视

改了规则，到底变好了还是变差了？哪些 Badcase 被修复、哪些是新引入的？没有版本管理与前后对比，调优变成「凭感觉」。

一句话痛点

如果一个辅助优化工具本身就需要疯狂复制粘贴、人肉归因，业务人员是绝对不会用的。本系统的存在意义，就是把这条链路彻底缩短。

02系统定位与目标

明确「是什么」「不是什么」，划清与新旧平台的边界。

2.1 系统定位

一个独立于内外网主干业务的「规则实验室」

本系统是 Python 辅助工具（外网工业级 Demo），不直连新旧平台。它接收脱敏的业务评测集与 Trace 日志，自动完成 Badcase 归因，并逆向生成/优化结构化的 BOM 配置，最终在看板中展示调优前后的核心指标对比。生成的规则专供上游抽取使用，与下游数据监控严格解耦。

2.2 核心目标

↓

极简输入

业务只上传 Excel 评测集，必要时补 chunk；不碰 IT 概念。

◎

智能归因

Badcase 自动归类到 5 类原因，给出可执行的调优动作。

◻

白盒输出

输出结构化 BOM 与 delta，业务看得懂、改得动、可采纳。

↻

闭环评估

版本化管理，前后对比，回归一目了然。

2.3 与新旧平台的关系

维度说明
连接方式物理解耦，不通过 API 对接；通过「业务手动录入 / 导出文件」同步 BOM 与跑分结果
BOM 源头本工具持有本地 BOM 规则库作为正式版源头，平台为消费方
数据合规业务在工具内可见真实数据；仅在数据出网去外部大模型时做假名化脱敏
提示词可见性新平台 Prompt 对用户不可见；本工具只操作 BOM 配置，不产出/不依赖完整提示词

03用户角色与典型场景

工具为不懂 IT 的合同商务人员而设计。

3.1 主要用户

人

合同商务部

定义条款/要素、维护评测集、判断抽取对错。不懂代码，熟悉合同业务语言。

⚙

业务实施人员

负责把工具产出的 BOM 录入新平台跑批、把跑分结果与 Trace 灌回工具。

3.2 三个典型场景

  1. 首版 BOM 冷启动——新条款/要素上线，用标注评测集一键生成初版定义、规则、画像与正反例。
  2. Badcase 调优——把 Trace + Badcase 粘进工具，自动归因并给出优化建议，采纳后产出新版 BOM。
  3. 版本回归对比——录入新旧两版跑分，看哪些 Badcase 被修复、是否引入新问题。

04核心业务流程（端到端闭环）

从评测集到评估看板，一条闭环；脱敏作为横切闸门守在出网环节。

①

数据加载

业务上传 Excel 评测集（自足），可选补充 chunk 关联原文。

输入 Excel / Chunk  →  产出 标准化测试集（含条款-要素父子结构与上下文）
▼
②

BOM 智能生成（核心引擎）

基于语义名称 + 标注测试集，逆向生成业务定义、抽取规则、召回画像、正反例。

输入 测试集 + 语义名称  →  产出 初版 BOM（V1）
▼  业务手动把 BOM 录入新平台跑批
③

跑分回收

业务把上一版与新版 BOM 的跑分结果录入工具；若有 Badcase，连同 Trace 一起带回来。

输入 跑分结果 + Trace + Badcase
▼
④

诊断与优化

对 Badcase 自动归因到 5 类，给出根因与结构化优化建议（BOM delta）。

输入 Trace + Badcase + 当前 BOM  →  产出 归因结果 + BOM delta
⬗ 横切闸门 · 数据脱敏：环节④ 在调用外部大模型前，自动对合同原文/提示词做假名化（【采购方】【供应商】【金额_1】…），业务在工具内仍看真实数据。
⑤

应用与导出

业务确认采纳优化建议，工具把 delta 应用到本地 BOM 库，产出新版本并导出。

输入 BOM delta + 当前 BOM  →  产出 新版本 BOM（V2…）+ 导出文件
▼
⑥

评估看板

按条款粒度展示业务指标，版本 Diff，标注哪些 Badcase 被修复、哪些是新引入。

输入 跑分结果 + BOM 版本  →  产出 指标看板 + 回归对比

05关键业务概念

统一术语，是业务与技术对话的基础。

术语业务含义
语义块即「条款」。父级单位，如「押金条款」。其下挂多个语义项。
语义项即「要素」。子级单位，挂在语义块下，如「保证金金额」「缴纳期限」。
BOM语义 BOM 定义 = 业务定义 + 抽取规则（核心匹配/绝对拦截）+ 召回画像 + 正反例。工具的核心产物。
召回画像新平台特有：正向关键词、易混淆词、语义查询、章节提示、条款/要素别名。决定「能不能把原文召回到上下文」。
Trace调用链路日志。含合同原文窗口（chunk）、模型推理（reasoning）、抽取状态（status）。
Badcase抽取错误样例。分两类：误抽（期望为空、抽取非空）、漏抽（期望非空、抽取为空）。
正反例用于 few-shot 的样例：正例 = 该抽的原文片段；反例 = 不该抽但易混淆的片段。
BOM delta一次优化产生的增量改动（新增关键词/拦截规则等），可应用到本地 BOM 库生成新版本。

06四大业务能力

每个能力都对应一个可独立使用、可单独评审的业务模块。

6.1 BOM 智能生成（核心引擎）

这是系统的核心发力点：输入语义名称 + 标注评测集，自动产出结构化 BOM。

  • 上下文来源（按抽取方式）：基于语义块抽取时，子要素上下文 = 父条款的期望结果（Excel 自足）；独立抽取时，上下文 = 全文/章节，需上传 chunk。
  • 补充入口：用户可自行上传 chunk 块，按文档 ID / 语义块编码关联，作为合同原文。
  • 易混淆词生成策略：先由正例推断出基线（永远有产出），当用户提供 chunk / 出现真实 Badcase 时，再用负例沉淀强化——chunk 不强制，但有则更准。

6.2 Badcase 诊断与优化

用一个「沉浸式大输入框」承接业务从新平台复制的整段 Trace，工具自动剥离出原文窗口、reasoning、抽取状态，再调用外部大模型完成归因与优化。

  • 智能粘贴：自动正则解析，不让业务填十几个框。
  • 强制结构化输出：大模型只返回 JSON（归因类别 + 根因 + BOM delta），严禁返回完整提示词。
  • 一键采纳：业务确认后，delta 应用到本地 BOM 库，产出新版本。

6.3 数据脱敏（出网合规闸门）

  • 定位：纯粹的出网合规闸门，不改变工具内数据展示——业务在工具内看真实数据。
  • 策略：假名化（同类角色替换），而非简单打星号，保留句法结构，让大模型能分析主谓宾。
  • 占位符：角色化、人可读——【采购方】【供应商】【金额_1】【人名_1】，返回的规则本身通用可用。
  • 规则可配置：对接公司合规清单（公司名、金额、人名、联系方式…）。

6.4 评估看板与版本管理

不用 AI 评测指标（Precision/Recall 业务听不懂），改用业务语言。

  • 业务指标：命中率（查全）、误判率（错杀）、规则拦截成功率。
  • 版本 Diff：V1（基线）vs V2（AI 调优后）对比，明确「修复了 N 个，引入了 M 个」。
  • 归因分布：饼图展示当前 Badcase 的 5 类原因占比，直观体现优化重心。

07错误分类体系（5 类）

对齐新平台真实分类，每类对应一条明确的调优动作路径。

召回问题

chunk 没进上下文

原文窗口里根本没看到目标文本，模型无从抽取。

调优动作补召回画像（正向关键词 / 语义查询 / 章节提示 / 别名 / 易混淆词），不动 BOM 规则
混合问题

多原因混杂

召回与 BOM 都参与了，归不到单一层级。

调优动作召回画像 + BOM 规则都要改
BOM 问题

定义/规则有误

原文在窗口里：过严导致漏抽，过松导致误抽。

调优动作改 BOM 定义 / 核心匹配规则 / 绝对拦截规则
Prompt 模板待优化

拼装的模板有缺陷

自动拼装的 Prompt 模板本身不够好。

调优动作调整 Prompt 模板配置
大模型推理问题

模型 reason 分析错

规则与召回都对，是大模型自己推理出错。

调优动作补 few-shot 正反例纠偏，不动规则/模板
为什么要分这么细

不同类别走不同动作。若把「大模型推理问题」误判成「BOM 问题」，工具会诱导业务去改本就正确的规则，越改越乱。精确归因 = 精确动作 = 不破坏既有正确性。

08业务价值与收益

工具为业务省下的，是时间、是门槛、是不确定性。

极简Excel 上传即用，消灭逐条复制粘贴
白盒Badcase 自动归因，从「猜」到「看」
闭环版本化回归，改好改差一目了然
收益维度具体表现
收敛调优范围从阅读/修改几千字 Prompt，缩小为只维护结构化 BOM 规则。
防止灾难性遗忘规则按序号与优先级隔离，改一处不影响全局，避免「牵一发而动全身」。
自带溯源校验所有结果可关联到源 chunk 与归因，杜绝模型编造。
降低使用门槛合同商务人员无需懂 IT、懂 Prompt，也能高质量调优。
