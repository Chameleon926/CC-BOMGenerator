# 规则编排智能体 · 开发前 PoC

## 目的

验证一个核心假设（开发前最高优先级）：

> **LLM 拿着「旧 BOM 规则 + Badcase」，能否归纳出更准的新规则？**

只验证"脑子"（LLM 归纳推理能力），**不造"躯干"**——抽取与准确率交给**真实新/旧平台手工跑**（平台自带准确率，天然金标准）。

## KISS 原则

- ❌ 不写复杂解析算法、不接 DB、不做脱敏、不做抽取器
- ✅ 只做：组装 Meta-Prompt → 调大模型 → 输出【归因 + 新 BOM】

## 三步走

### Step 1 · 准备黄金切片 `data/sample_slice.yaml`

挑 **1 个常出错、具代表性**的条款，填四块：
`当前 BOM` + `正例(3~5)` + `漏抽反例(3~5)` + `误抽反例(3~5)`。
涉密数据手工打码（公司名→XXX、金额→N元）。每类 3~5 条即可。

### Step 2 · 跑规则智能体

```bash
cp .env.example .env            # 填 LLM_API_KEY / LLM_BASE_URL / LLM_MODEL
pip install -r poc/requirements.txt
python poc/run_rule_agent.py poc/data/sample_slice.yaml
```

输出落在 `poc/output/new_bom_<时间>.md`。**重点看两件事**：
1. 归因对不对？（有没有说到点子上、落到 5 类里的哪类）
2. 规则有没有**过拟合**？（写死具体公司名/金额 = 坏；提炼成"供应商主体"这类泛化特征 = 好）

> 想纯手工也行：把下方 [Meta-Prompt 模板] + 切片数据粘进任意对话框（Gemini/Claude/ChatGPT）。

### Step 3 · 手工闭环验证（准确率）

把生成的新 BOM **配置进新/旧平台** → 拿一批**没给大模型看过的**测试数据（hold-out）跑 → 看平台准确率：
- 漏抽减少了吗？有没有引入**新误抽（回归）**？
- **净改善 > 0** → 核心通路成立，可理直气壮进入正式开发。
- 净改善 ≤ 0 → 及早暴露"LLM 调规则没那么靠谱"，省掉后面所有基建的浪费。

## 它同时验证两件事

1. LLM 能否生成更好的规则（脑子）—— 主目标
2. 我们的 **5 类归因体系**（召回/混合/BOM/Prompt模板/大模型推理）是否好用 —— 顺带验证

## 模型配置（预留）

模型完全可配置，自行填 `.env`：
- `LLM_API_KEY`：你的 api-key
- `LLM_BASE_URL`：端点地址（OpenAI 兼容；Gemini 用 `https://generativelanguage.googleapis.com/v1beta/openai`，自建/代理填对应地址）
- `LLM_MODEL`：模型名

正式工具的"模型配置"页就是从这套配置长出来的。

## 附：Meta-Prompt 模板（手工用）

```
你是一位资深法务科技工程师，精通合同语义分析与大模型抽取规则（语义 BOM）配置。
任务：分析当前「抽取规则」在真实数据上的错误（漏抽/误抽），归纳出更精准的新规则。

【目标条款】<条款名>
【当前业务定义与规则（旧 BOM）】<粘贴旧规则>
【正例：当前规则抽对的原文】<...>
【漏抽反例：本该抽却没抽的原文】<...>
【误抽反例：本不该抽却错抽的原文】<...>

请完成：
1) 错误归因诊断：从【召回/混合/BOM/Prompt模板/大模型推理】5 类里判断主因，并说明为什么。
2) 提炼新规则（必须泛化，禁止针对具体公司名/金额/人名写规则，防过拟合）：
   【新增正向关键词】【新增易混淆词】【新增/修改 绝对拦截规则】【新增/修改 核心匹配规则】
要求：自然语言逻辑，业务人员能看懂；不要代码、不要正则。
```
