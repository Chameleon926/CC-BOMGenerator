## 1. 仓库与分支策略

两个分支是整个协作的核心。

### 1.1 分支结构

```
main                  ← 稳定集成分支。只通过 PR 合并，不直接推
  ├─ feature_lindayu  ← 林大雨的个人开发分支
  └─ feature_yangli   ← 杨力的个人开发分支
```

### 1.2 工作流

**个人开发**：各自在 `feature_lindayu` / `feature_yangli` 上开发、commit、push，**不直接推 main**。

**合并到 main**：

```bash
# 先从 main 切一个短命分支
git checkout main
git pull
git checkout -b pr/diagnosis-node-v2
# 改代码、commit
git push origin pr/diagnosis-node-v2
# → 在 GitHub 上开 PR（pr/diagnosis-node-v2 → main）
# → 对方 review → 合入 main
```

**同步对方的工作到自己的分支**：

```bash
git checkout feature_yangli
git pull origin feature_yangli       # 拉取杨力最新代码
git checkout feature_lindayu
git merge origin/feature_yangli       # 合并到自己分支（有冲突则解决）
```

### 1.3 开工前必做

```bash
git checkout feature_lindayu
git pull origin main                  # 把 main 的最新内容拉进来
```

**只要对方可能改了共享文件（contracts、表结构、prompts），必须 `pull` 后再改**。

## 2. 契约先行（Contract-First）

这是并行开发的核心范式，绝对遵守。

### 2.1 节点管线

```
ingest(数据加载) → anonymize(脱敏) → generate(生成) → diagnose(诊断) → optimize(优化) → apply(应用) → evaluate(评估)
```

### 2.2 契约文件位置

```
src/cc_bom_generator/contracts/
├─ __init__.py
├─ test_set.py        # ingest 的输出
├─ bom.py             # generate/optimize 的核心产物
├─ diagnosis.py       # diagnose 的输出
├─ trace.py           # trace 解析后的结构化字段
└─ evaluation.py      # evaluate 的指标
```

### 2.3 规矩

- **动手实现任何节点前，先在该目录定义并提交该节点的输入/输出 Pydantic schema**，然后才能开始写实现。
- 契约就是协作边界：A 实现 `generate`、B 实现 `diagnose`，彼此只看对方的契约，不读对方实现内部。
- **改契约 = 开 PR 让对方 review**。契约变更属于"高影响变更"，必须显式同步，不能默默改。
- 所有接口字段必须写 `description=`，标明含义、单位、是否可为空。

## 4. 数据库与模型配置

### 4.1 MySQL

**本地库配置**——两人都在本地各跑一个 MySQL，数据互不可见且不入库：

```sql
CREATE DATABASE cc_bom_generator CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

**连接配置**：`config/llm.yaml` 里写 MySQL 连接（**已 gitignore，差异互不影响**）。

**Schema 管理**：用 **SQLAlchemy + Alembic**：

- 表结构变更必须通过 Alembic 迁移脚本来修改，**不准手动改表**。
- 迁移脚本提交到 git，对方 `alembic upgrade head` 即可同步结构（数据每人各自本地，完全不同步）。
- 共用模型文件放：
    ```
    src/cc_bom_generator/db/
    ├─ models.py          # SQLAlchemy 模型
    └─ versions/          # Alembic 迁移脚本（由 alembic 自动管理）
    ```

### 4.2 模型配置

```bash
cp config/llm.example.yaml config/llm.yaml    # 各自填各自的
```

已有字段：

```yaml
api_key: "sk-..."           # 必填
base_url: ""                # 留空=默认端点
model: "gpt-4o-mini"        # 必填
temperature_stage1: 0.2     # 定义+规则（收敛）
temperature_stage2: 0.5     # 召回画像（发散）
mysql_host: "127.0.0.1"     # MySQL 连接
mysql_port: 3306
mysql_user: "root"
mysql_password: ""
mysql_database: "cc_bom_generator"
```

> ⚠️ 注意：`config/llm.yaml` 已 **gitignore**，内含 api-key、model、数据库密码，绝不入库。

### 4.3 内网模型 vs 外网模型

支持在 UI 切换（`config/llm.yaml` 配两套或 UI 里选）。外网路径**自动执行脱敏**后方可出网；内网路径不脱敏。

## 5. 代码约定

### 5.1 通用

- 文件编码：UTF-8
- 标识符用英文，注释与业务文档用中文
- 每条公开函数/方法必须写 docstring（功能、参数、返回）
- 不要超过单文件 500 行，超了拆模块

### 5.2 提示词管理

**提示词与代码严格分离**。所有 prompt 放在 `prompts/` 目录下，用纯文本 + `{{var}}` 占位符，代码只负责 `load + render`：

```python
# ✅ 正确做法
prompt = (Root / "prompts" / "gen_stage1.txt").read_text()
messages.append({"role": "user", "content": render(prompt, clause="xxx", cands="...")})
```

```python
# ❌ 错误做法 —— 以下代码禁止出现
prompt = "你是一位资深……【目标条款】{clause}"
```

prompts 目录结构：

```
prompts/
├─ system.txt               # 系统人设
├─ gen_stage1.txt           # generate：定义+规则
├─ gen_stage2.txt           # generate：召回画像（派生）
├─ opt_stage1.txt           # optimize：定义+规则（+诊断）
├─ opt_stage2.txt           # optimize：召回画像（派生）
└─ verify.txt               # Stage3：规则自检
```

### 5.3 测试

- 每个节点必须有单元测试（`tests/test_<node>.py`）
- 给定输入契约 → 断言输出契约字段正确，不测编排
- run：
  ```bash
  pytest tests/ -v
  ```

## 8. 争议仲裁

遇到分歧，按以下优先级解决：

1. 设计的核心原则（功能拆解里的可信度分层/契约先行/解耦）
2. 已有设计文档中的约定
3. 谁改的代码谁有最终解释权，除非改坏了契约
4. 意见严重不一致时，先按更保守的方向走，再找上级决策
