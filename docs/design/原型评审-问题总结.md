# 同事 UX 原型评审 · 问题总结

> 评审对象：`docs/design/prototype.html`（feature_yangli，交互原型）
> 评审视角：① 合同商务部业务用户 ② 数据 / 合同质量风险专家
> 结论：**视觉 8 分，产品架构 5 分**——是"功能堆得全的工具箱"，不是"一条顺的业务流"。

---

## 一、整体判断

产品灵魂是「**规则编排**」，但 UI 把它拆成了若干**并列的独立工具**。业务进来面对 9 个入口，第一反应是"我该点哪个"。核心矛盾：**信息架构是工程师视角，不是业务视角**。

---

## 二、业务用户视角问题

| # | 问题 | 说明 |
|---|---|---|
| 1 | **入口太多太散** | 左侧 9 个导航（条款库/测试集洞察/辅助生成/辅助优化/文档脱敏/数据脱敏/提示词拼装/设置/详情）。业务心智是"一个条款从生到优"，不是 9 个工具。 |
| 2 | **两种场景被割裂** | "辅助生成"与"辅助优化"分两个入口，但本质是**同一条款生命周期的两步**（生成→跑→优化），分开 = 业务来回跳页面。 |
| 3 | **上传入口到处都是** | 辅助生成、数据脱敏、文档脱敏都能上传——同一文件满世界传，无"绑定条款"概念。 |
| 4 | **术语劝退业务** | `TestSetProfile`、语义去重置信度、`\d{1,3}(,\d{3})*` 正则、Embedding——业务一眼关。 |
| 5 | **"提示词拼装"是工程师玩具** | 手填四项拼模板，与"辅助优化"重叠，业务用不来，建议删除/并入优化。 |
| 6 | **"测试集洞察"突兀** | 业务来是**编规则**的，不会主动看测试集数值分布饼图；它不该是一级导航。 |

---

## 三、数据运营问题（重点，当前太浅）

当前"数据运营" = 测试集洞察（去重率 / 分布 / 格式模板正则 / 触发词 / 难例）。
**这是"数据工程指标"，不是"领导 / 质量风险关心的指标"**，一眼看不懂——因为它**没回答任何一个业务/领导会问的问题**。

### 领导 & 合同质量专家真正关心的 6 个问题（当前一个没答）
1. **整体抽取健康度**：N 个条款，整体准确率？红/黄/绿分布？趋势？
2. **风险条款在哪**：哪几个准确率最低 / 下降最快？（领导：风险在哪、要不要投人）
3. **优化投入产出**：这轮修了几个 badcase、引入几个回归、净收益、谁改的、何时？
4. **灾难性遗忘**：改 A 条款规则，B 条款准确率掉没掉？（质量专家最怕）
5. **系统性瓶颈在哪层**：badcase 5 类归因占比（召回/BOM/推理/Prompt/混合）→ 决定资源投向。
6. **测试集够不够好**：数据不均/缺场景（**仅准确率低时触发**，给"补哪些样本"建议，反哺优化）。

### 结论
- 数值分布/正则模板/触发词是**单条款数据画像**，应塞进"条款详情"做配角，不占一级导航当"数据运营"。
- 数据运营要重做成 **「质量与风险大盘」**，每块都指向"发现问题→行动"。

---

## 四、上传 & 脱敏入口问题

- 上传分散在多个模块，无条款绑定。
- 脱敏作为独立工具页（文档脱敏 + 数据脱敏两个），但它本质是"**数据出网前的自动步骤**"，不该是独立手工工具。
- 用户设想（正确）：**一个上传入口绑定条款**；每条款挂三类数据：初始测试集 / 优化测试集(跑后) / trace，对应两种场景；辅助生成时"选条款+选数据→自动判场景→查旧BOM（有则优化、无则生成）"。

---

## 五、信息架构问题

9 个并列入口 → 业务认知负担重、旅程断裂。应按"**一条业务流**"收敛。

---

## 六、重构方向（三件套）

| 现在（9 入口，散） | 重构后（3 入口，顺） |
|---|---|
| 条款库 + 辅助生成 + 辅助优化 + 详情 | **① 条款工作台**（列表→选条款→生成/优化/详情，三场景合一） |
| 测试集洞察 | （降级：进工作台详情 + 质量大盘下钻） |
| 文档脱敏 + 数据脱敏 + 提示词拼装 | （脱敏自动化进设置；拼装并入优化；删独立入口） |
| — | **② 质量大盘**（健康度 + 风险 + 收益 + 瓶颈） |
| 设置 | **③ 设置**（脱敏规则 + 模型配置） |

**业务旅程变一条线**：工作台选条款 → 一键生成（三阶段：定义+规则→画像→自检）→ 拿去跑 → 回填结果一键优化 → 质量大盘验收。

## 八、多轮对话 Agent（建议留到后期）

- **先别做 chat 窗口**。当前三阶段（定义+规则→画像→自检）本身已是"无对话的 Agent 流"，先把工作台跑顺。
- 后期做成**「条款工作台」侧边 Copilot**（非独立模块），场景：自然语言优化（"把违约责任漏抽列出来"）、解释答疑（"为什么准确率掉了"）、批量优化。
- 技术选型：**LangGraph**（状态机 + 人审打断，最贴合多阶段+要人确认）；记忆=历史修复经验（向量库，Phase 2）。对话是加速器，不是地基。
